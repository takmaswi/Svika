# R4.5 — feat(R4.5): kombis follow the road (sub-segment interp + densify warnings)

> Paste this whole document into Claude Code as one job. Same operating frame as R5: Auto Mode within `.claude/settings.local.json` allowlist, conventional commits direct to `main`, narrow staging, `pnpm typecheck && pnpm lint && pnpm build` after edits, no push until Cowork greenlights the rehearsal screenshots.

## Context

R4.5 fixes a long-standing visual bug: kombi markers move in straight chord lines between road corners instead of following the actual road polyline. Before R4.5 the four routes were 3–9-point hand-traces; the densifier silently fell back to raw because the old Mapbox token had URL restrictions. As of 2026-04-29 the token has been rotated, `pnpm db:seed` has been re-run, and the four routes now have road-snapped polylines (354 / 199 / 259 / 61 coordinates instead of 9 / 9 / 5 / 3).

R4.5 has two fixes:

- **Fix A — sub-segment interpolation.** Use `pointAtDistance(routePolyline, lerpedMeters)` from `lib/sim/geometry.ts` to position kombis along the actual road polyline, instead of the current chord-based lerp on `[lat, lng]`. Pre-densify this looked roughly OK because the polyline was 9 corners; post-densify it's 354 corners and the chord lerp would visibly cut across blocks.
- **Fix B — densify visibility.** Add `console.warn` to all four fallback paths in `lib/mapbox/densify.ts` and a "kombis may go off-road" warn in `lib/sim/simRunner.ts` when source === "raw". Today the fallback is silent and we couldn't tell at a glance whether densification had succeeded.

R4.5 is a small surgical change (~5 edits across 3 files). No business logic. No palette work. No new components.

## Locked invariants (do not drift)

- `[lat, lng]` is the convention inside `lib/sim/geometry.ts` and inside `simRunner.ts`. `[lng, lat]` is the convention inside GeoJSON and Mapbox sources. The flip happens at `lib/network/loadNetwork.ts:67`. Do not move the flip; do not change the ordering inside geometry.ts.
- `KombiTickPayload` is the broadcast wire format. Adding a field here is a breaking change for any subscriber — but the only subscriber is `PassengerMap.tsx`. Still, treat it as a contract change and update both ends in the same commit.
- The RAF loop already exists. Do not introduce a second RAF; extend the existing one.
- Graceful fallback: if `routePolylines.get(routeId)` returns undefined (e.g., a route the sim broadcasts that the network seed didn't include), keep the existing chord lerp as a fallback. Do not throw.
- Phase marker for prod-verify: add `data-phase="r45"` to the PassengerMap container `<section>` element (the outermost wrapper in the component's return). One attribute, used only as a string sentinel to confirm the build shipped.

## Step 1 — `KombiTickPayload`: add `progressMeters`

In `lib/sim/simRunner.ts`, extend the `KombiTickPayload` interface (currently lines 44–53):

```ts
export interface KombiTickPayload {
  vehicle_id: string;
  route_id: string;
  lat: number;
  lng: number;
  direction: "outbound" | "inbound";
  /** Compass bearing (0–359, clockwise from north) — drives icon-rotate. */
  bearing: number;
  /** Distance along the densified route polyline, in meters from start.
   *  Drives sub-segment road-following interpolation in PassengerMap RAF.
   *  Same value the sim uses internally to step the kombi via advance(). */
  progressMeters: number;
  at: string;
}
```

Then find where the sim ticks each kombi (look for the loop that calls `advance()` or similar around line 80–120). At each tick, the sim already knows the cumulative distance along the polyline — that's how `advance()` produces the next position. Surface that distance into the broadcast payload.

If the sim stores per-vehicle progress in a Map or in the `vehicles` row, read it there. If it's a local variable inside the tick loop, just include it. The exact name in the sim is whatever the existing `pointAtDistance` call uses — grep for `pointAtDistance` inside `simRunner.ts`.

The wire format is now contract-bound. PassengerMap will read `t.progressMeters` and break if it's missing. Make sure every code path that builds a `KombiTickPayload` populates it. (There should be one builder.)

## Step 4 — `routePolylines` cache + RAF loop swap

At PassengerMap module load (or inside the constructor effect, before the broadcast handler binds), build a `Map<string, ReadonlyArray<readonly [number, number]>>` keyed by `route_id`:

```ts
// PassengerMap component scope, after `network` is in hand:
const routePolylines = useMemo(() => {
  const m = new Map<string, ReadonlyArray<readonly [number, number]>>();
  for (const route of network.routes) {
    // network.routes[*].geometry.coordinates is GeoJSON [lng, lat][].
    // pointAtDistance wants [lat, lng][]. Flip on the way in.
    const flipped: Array<readonly [number, number]> = route.geometry.coordinates.map(
      ([lng, lat]) => [lat, lng] as const,
    );
    m.set(route.route_id, flipped);
  }
  return m;
}, [network]);
```

> If `network.routes[*].geometry` is null/undefined for any route (shouldn't be after seeding, but defensively), skip it. The fallback in the RAF loop will catch missing keys.

Then in the RAF loop (around line 1471–1476), replace the existing chord lerp body. It currently looks something like:

```ts
// before — chord lerp on lat/lng
const t = clamp((now - entry.broadcastAt) / TICK_MS, 0, 1);
const lat = entry.prev[0] + (entry.next[0] - entry.prev[0]) * t;
const lng = entry.prev[1] + (entry.next[1] - entry.prev[1]) * t;
const bearing = lerpBearing(entry.prevBearing, entry.nextBearing, t);
```

Replace with progress-aware lerp:

```ts
// after — progress-aware road-following lerp
const t = clamp((now - entry.broadcastAt) / TICK_MS, 0, 1);
const lerpedMeters =
  entry.prevProgressMeters + (entry.nextProgressMeters - entry.prevProgressMeters) * t;

const polyline = routePolylines.get(entry.routeId);
let lat: number;
let lng: number;
if (polyline && polyline.length >= 2) {
  const point = pointAtDistance(polyline, lerpedMeters);
  lat = point[0];
  lng = point[1];
} else {
  // Fallback: chord lerp (same as before R4.5).
  lat = entry.prev[0] + (entry.next[0] - entry.prev[0]) * t;
  lng = entry.prev[1] + (entry.next[1] - entry.prev[1]) * t;
}

const bearing = lerpBearing(entry.prevBearing, entry.nextBearing, t);
```

Import `pointAtDistance` at the top of `PassengerMap.tsx`:

```ts
import { pointAtDistance } from "@/lib/sim/geometry";
```

> The fallback to chord lerp matters: if a kombi tick comes through for a route the seed doesn't carry (shouldn't happen, but defensive), or `pointAtDistance` returns NaN for some edge case (lerpedMeters past polyline end), we don't want the kombi to disappear. Chord lerp keeps it on screen.

## Stop conditions — surface and ask, do not improvise

- The sim runner has more than one `KombiTickPayload` builder — adding `progressMeters` to one but not all of them will silently produce broken broadcasts. Grep for everywhere `vehicle_id:` and `bearing:` co-occur in `simRunner.ts` and audit each.
- `pointAtDistance` returns `NaN` for some inputs at the polyline boundaries — surface; the brief expects it to clamp gracefully but if it doesn't the chord-lerp fallback needs to also catch NaN.
- `network.routes[*].geometry.coordinates` is missing for a route — densify must have fallen back to raw despite the rotation. Check `densify.ts` warnings and `pnpm db:seed` output.
- `pnpm build` fails because `pointAtDistance` import path differs from `@/lib/sim/geometry` — the path alias is set in `tsconfig.json`; if it doesn't resolve, surface.
- The kombi marker visibly stutters or jumps after the change (RAF loop runs at the wrong cadence) — surface; this means the meters/seconds calibration in the lerp is off and we need to re-derive from the sim's tick interval.
- Otherwise run all the way through to commit + push + BUILD-LOG.

End of brief.
